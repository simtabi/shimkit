## Why doc-sync as its own release

shimkit's release notes are user-facing — the README + per-tool
docs are how someone evaluating the project decides whether to
install it. Letting them drift behind the shipped surface is a
slow-motion credibility leak. Cutting a tag for the sync makes
the diff auditable and gives the GitHub Release page a clear
"this is the version that has accurate docs for everything
shipped" anchor.

The bundled `config/shimkit.schema.json` is the file `$schema`
references in user configs point at — keeping it regenerated
means VS Code / IntelliJ autocompletion works for the new
`tls.dns-route53` + `framework.symfony` + `framework.django`
blocks without users having to wait for the next feature release.
