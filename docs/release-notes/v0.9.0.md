# shimkit 0.9.0

`shimkit db --on-host` — charter-completion release: opt-out from
container-first on `shimkit db`. Last of the deferred v0.5+
candidates. For the full machine-readable changelog, see
[`CHANGELOG.md`](../../CHANGELOG.md).

---

## TL;DR

```
shimkit db <engine> up     --on-host     # MODERATE
shimkit db <engine> down   --on-host     # MODERATE
shimkit db <engine> status --on-host
shimkit db <engine> shell  --on-host
```

When `--on-host` is set, `up`/`down`/`status` route through
`shimkit.core.HostService` (systemd on Linux, brew services on
macOS) and manage an *already-installed* engine rather than a
container. `shell` shells out to the host's CLI client
(`mysql -h 127.0.0.1`, `psql -h 127.0.0.1`) — exporting
`PGPASSWORD` for postgres so the existing password flow works
unchanged.

Available for mysql / mariadb / postgres only. mongo and
phpmyadmin are intentionally container-only.

---

## Security shape

This is the **audit-completion bit** for the v0.5.0 migration:

> shimkit DOES NOT install packages in --on-host mode. If the
> engine's client binary isn't on PATH, the command refuses
> (EX_UNAVAILABLE for shell, EX_FAIL for up/down).

The original ubuntu scripts had five Critical security flags from
install-on-host patterns (0.0.0.0 binds, deprecated apt-key,
curl|sh). `--on-host` explicitly avoids reproducing any of them.
Users who want mysql on the host install it themselves via apt /
brew / dnf; shimkit just manages it.

## What's new

### `--on-host` flag on db up / down / status / shell

```bash
# Linux: systemctl start mysql
shimkit db mysql up --yes --on-host

# macOS: brew services start postgresql@16
shimkit db postgres up --yes --on-host

# Read-only state report
shimkit db mysql status --on-host

# Connect via host client
shimkit db mysql shell --on-host
shimkit db postgres shell --on-host --password secret    # PGPASSWORD-passthrough
```

### `shimkit.core.HostService`

New cross-platform service-manager facade.
`HostService.detect(platform)` returns:

- `SystemdHost()` on Linux (delegates to the existing
  `shimkit.core.Systemd` facade)
- `BrewServicesHost()` on macOS (`brew services list/start/stop`)
- `None` on Windows or unsupported systems

Each subclass implements `state()` / `start()` / `stop()` returning
a typed `HostServiceResult(ok, state, stdout, stderr)`.

### `tools.db.host_services` config

Per-engine service-name mapping. Defaults:

| Engine    | Linux         | macOS              |
|-----------|---------------|--------------------|
| mysql     | `mysql`       | `mysql`            |
| mariadb   | `mariadb`     | `mariadb`          |
| postgres  | `postgresql`  | `postgresql@16`    |

Override per-install when your distro or homebrew formula diverges.

### Engine ABC additions

Three new methods on the per-engine driver:

- `supports_on_host()` — capability flag
- `host_shell_argv(password)` — host-side CLI argv (`-h 127.0.0.1`)
- `host_client_binary()` — the binary name (`mysql` / `psql`)

mysql / mariadb / postgres override `supports_on_host=True` and
provide the argv; mongo / phpmyadmin keep the default `False`.

## Manager safety

`DbManager.boot(on_host=True)` skips the docker preflight entirely
— useful for users who never installed Docker. Container-mode
methods now assert `self._env is not None` at entry so a stray
container call on an on-host manager fails loudly rather than
`AttributeError` on None.
