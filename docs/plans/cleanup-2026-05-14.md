# Post-v0.2.0 cleanup plan

End-of-session audit findings and the prioritized work to close
them. Frozen at `2026-05-14`; reflects the state at commit
`53782f8` (20 commits ahead of `Initial release`).

The brief in [`prompt.md`](../../prompt.md) defined a substantial
surface; this file is the honest accounting of what shipped vs. what
the brief asked for.

Items are P1 (real-functional gap against shipped docs), P2 (test
coverage gap), or P3 (low-risk polish).

## P1 — Items that contradict shipped documentation

### 1.1 Shared CLI flags partially wired

**Promised** (in `docs/architecture.md`,
`docs/onboarding.md`, `prompt.md` Phase 2 CLI standards block,
`docs/release-notes/v0.2.0.md`): every new subcommand accepts a
uniform set of flags — `--dry-run`, `--json`, `--quiet`,
`--verbose`, `--yes`, `--force`, `--log-file`, `--color={auto,
always,never}`, `--no-color`, `--timeout`, `--no-input`.

**Delivered**: `--dry-run`, `--json`, `--quiet`, `--verbose`,
`--log-file` are wired across the three tool subapps.
`--timeout` is wired only on `adguard verify`.
`--yes`, `--force`, `--no-color`, `--no-input`, and `--color` are
defined as singletons in `src/shimkit/core/cli_flags.py` but never
imported by any command.

Verified at HEAD:

```bash
$ shimkit dns flush --yes
No such option: --yes
$ shimkit dns diagnose --no-color
No such option: --no-color
```

**Fix options (pick one):**

- **(a) Finish wiring.** Import the missing flags in every
  `commands.py`; have the `_bootstrap()` helper consume `--quiet`/
  `--verbose`/`--no-color` and call into `UI.set_quiet(...)`,
  `set_verbose(...)`, and a new `UI.set_color(...)` respectively.
  Severe-tier commands ignore `--yes`. Moderate-tier commands honour
  `--yes` to skip the `[y/N]` prompt (see 1.2). Estimated: 30–50
  lines across three `commands.py` files plus a `UI.set_color()`
  helper.

- **(b) Trim the surface.** Delete the unused singletons from
  `cli_flags.py`; update `architecture.md`, `onboarding.md`,
  `prompt.md`, and `release-notes/v0.2.0.md` to claim only the five
  that actually work. Estimated: 20 lines deleted, four doc edits.

**Recommendation**: **(a)** — the flags were promised to users in
release notes and they're cheap to wire. Reserve **(b)** as a
fallback only if `(a)` turns out to be much larger than expected.

### 1.2 MODERATE-tier `[y/N]` prompts not implemented

The brief defined three confirmation tiers: MILD (proceed),
MODERATE (`[y/N]` prompt; `--yes` bypasses), SEVERE (typed
`--confirm <token>`). SEVERE is wired (`dns reset`, `dns fix
--nuclear`, `docker-clean nuke`). MODERATE is not.

Commands the brief listed as MODERATE that currently execute
without prompting:

- `dns set <ip>...`
- `adguard ports set --dns N --http N`
- `docker-clean prune-images / prune-volumes / prune-networks / prune-builders`

**Fix**: add a `prompt_confirm()` helper to `core/menu.py` that
returns False on `--no-input` or non-TTY stdin, returns True on
`--yes`, otherwise asks `[y/N]` via `Menu.confirm()`. Wire into the
listed commands. Estimated: 20 lines + per-command wiring.

## P2 — Test-coverage gaps against the brief's "mandatory minimum"

### 2.1 No test for extras-missing → exit 69

The brief required: *"Boot exits 69 when the optional extra is
missing."* I added platform-gate exit-69 tests but never the
extras-missing variant. Both `tools/adguard/manager.py` and
`tools/docker_clean/manager.py` have a `_require_optional_extras()`
helper that's called from `boot()`; neither is exercised by any
test.

**Fix**: monkeypatch `builtins.__import__` to fail on the extra's
module name; assert `boot()` raises `SystemExit(69)`. Two tests, one
per tool. Estimated: ~30 lines.

### 2.2 No tests for `--quiet`, `--log-file`, `--verbose` behaviour

Grep returns zero hits for these flag names in `tests/`. Each one is
a wiring contract:

- `--quiet` should call `UI.set_quiet(True)` and suppress non-error
  UI output.
- `--log-file PATH` should call `attach_file_handler(PATH)` and
  cause one JSONL line per `_LOG` call.
- `--verbose` should call `set_verbose(True)` and raise the root
  logger to DEBUG.

**Fix**: three CLI-level tests using `monkeypatch` of the
core/log/UI hooks. Estimated: ~50 lines.

### 2.3 Coverage to 85%

Current 66.07%; target 85%. Documented as deferred in
[`known-issues.md`](known-issues.md). Most of the remaining gap
lives in interactive `Manager.run()` menus and the most-destructive
paths.

**Status**: low priority unless `v0.3.0` brings new logic in those
files. No action this cycle.

### 2.4 Stale test counts in shipped docs

- `CHANGELOG.md` claims "216 tests" — reality at HEAD is 222.
- `docs/release-notes/v0.2.0.md` references the same older number.
- `docs/architecture.md` ends with a per-tool coverage table that

**Fix**: refresh the three count claims in one pass. Estimated: 3
text edits.

## P3 — Low-risk polish

### 3.1 EX_USAGE / EX_CONFIG defined in the brief but unused

The brief documents seven exit codes; in practice we use 0, 1, 2,
69, 77. EX_USAGE (64) and EX_CONFIG (78) never fire.

This is fine: EX_USAGE is what Typer emits automatically for
argument validation, and EX_CONFIG is implicitly used by config
validate (which currently exits 1). The brief's table was
aspirational. Either update the table to "0/1/2/69/77/130 in
practice; 64/78 reserved for future" or wire EX_CONFIG into
`shimkit config validate` failures. The latter is one-line.

**Recommendation**: wire EX_CONFIG into `config validate` failures.
Estimated: 1 line + test update.

### 3.2 Optional `gh attestation verify` smoke

Documented as a low-priority follow-up in
[`known-issues.md`](known-issues.md). No action this cycle.

### 3.3 Color singletons

`cli_flags.py` defines `NO_COLOR` but the brief listed
`--color={auto,always,never}` as the primary flag. Either ship
both or document the difference. Aligned with P1.1 above —
addressed there.

## What this audit is NOT

This is **not** a list of bugs in shipped behaviour. The shipped
code does what it does; the gaps are between what we **promised in
documentation** and what we **wired in code**. Users running
`shimkit dns flush` will see correct behaviour today — they just
won't be able to pass `--yes` to it. The gap is real but the
existing surface is sound.

Pulling these into shape closes the v0.2.0 release notes' promises
and brings the docs into alignment with reality. None of this work
is a release-blocker for a 0.1.x patch release, but P1.1 + P1.2 + P1.3
ARE release-blockers for `v0.2.0` if we want the release notes to
match the binary.
